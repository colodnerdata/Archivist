# Archivist Plan

## Purpose

This file tracks the current implementation state of the repository. The old
"first commit" plan is obsolete: Archivist already has a working CLI, later
pipeline stages, and a meaningful test suite.

This plan answers three questions:

1. What is implemented now?
2. What is verified now?
3. What should be improved next?

---

## Current Status

### Implemented pipeline

| Phase | Status | Notes |
|---|---|---|
| Phase 1 - Scan | Implemented | Recursive scan, resume support, exclusion rules, duplicate detection, inaccessible-path recording |
| Phase 2 - Triage | Implemented | Batched Ollama triage, duplicate auto-skip, CSV writeback |
| Phase 3 - Manual review | Implemented as workflow | Users edit `review` and `decision` columns directly in the CSV |
| Phase 4 - Summarize | Implemented | Inheritance-aware review gating, extractor-based content loading, text and image summarization |
| Phase 5 - Organize | Implemented | LLM-generated taxonomy plus `organized_path` assignment |
| Phase 6a - Copy | Implemented | Copies `KEEP` and `ARCHIVE` files, verifies copies, appends to kept-hash registry |
| Phase 6b - Manifest | Implemented | Generates delete manifest from effective decisions |
| Phase 6c - Delete | Implemented | Validates manifest against current CSV state, deletes files, logs results |
| Diagnostic - Resolve | Implemented | Shows effective inherited `review` and `decision` values for a path |

### Current repository modules

- `archivist.py`: CLI entry point and subcommand dispatch
- `scanner.py`: scan and duplicate detection
- `triager.py`: LLM triage and response parsing
- `inheritance.py`: effective-value resolution across path ancestry
- `extractor.py`: file-type-specific extraction and truncation helpers
- `summarizer.py`: reviewed-file summarization
- `organizer.py`: taxonomy proposal and organized-path assignment
- `executor.py`: copy, manifest, and delete actions
- `csv_utils.py`: safe CSV write helpers
- `llm_client.py`: Ollama connectivity and generation helpers

### CLI surface that exists now

The current CLI exposes these commands:

- `scan`
- `triage`
- `summarize`
- `organize`
- `copy`
- `manifest`
- `delete`
- `resolve`

## Verified Today

### Automated coverage present

Current tests cover these areas:

- `tests/test_scanner.py`: scan output, exclusions, resume behavior, duplicate hashing, inaccessible directories, progress and interrupt messaging
- `tests/test_triager.py`: LLM response parsing, validation, duplicate marking, prompt construction
- `tests/test_inheritance.py`: explicit values, inheritance behavior, blank handling, missing-column behavior
- `tests/test_extractor.py`: extraction behavior, unreadable and missing files, token truncation
- `tests/test_executor.py`: copy, kept-hash updates, manifest generation, inherited delete handling, stale-manifest rejection, matching-manifest deletion

### Verification gaps

These areas appear implemented but are not represented in the current test
files:

- `summarizer.py`
- `organizer.py`
- CLI argument handling and command dispatch in `archivist.py`
- end-to-end pipeline coverage across scan -> triage -> summarize -> organize -> execution

---

## Immediate Priorities

### 1. Add missing tests for later stages

Highest-value additions:

- summarizer eligibility and skip rules
- summarizer behavior for unreadable files and LLM failures
- organizer parsing and assignment writeback
- organizer empty-input behavior
- CLI smoke tests for each subcommand

Rationale: the biggest current risk is regression in implemented but
under-verified stages.

### 2. Add one narrow integration test for the CSV pipeline

Build a fixture-driven path that exercises:

1. scan output creation
2. triage writeback
3. manual `review` and `decision` setup
4. summarize output
5. organize output
6. copy or manifest behavior

Rationale: the CSV is the system contract. A single end-to-end test will catch
cross-stage schema drift better than isolated unit tests alone.

### 3. Harden CSV schema expectations

Make stage boundaries explicit:

- required columns per stage
- columns created by each stage
- normalization rules for booleans, decisions, and blank values

Rationale: silent CSV contract drift will break later stages in ways that are
hard to diagnose.

### 4. Tighten Ollama error handling and retry behavior

Focus on:

- clearer surfaced errors when configured models are missing
- better parse-failure diagnostics for summarize and organize responses
- consistent retry behavior across LLM-backed stages

Rationale: the main runtime instability is model I/O, not filesystem walking.

## Working Definition Of Done

The repository is in a strong maintenance state when all of the following are
true:

- all current subcommands are covered by at least one automated test path
- later stages (`summarize`, `organize`) have direct unit coverage
- at least one integration test exercises the full CSV pipeline
- README and `config.yaml` describe the actual runtime assumptions
- failures in filesystem access or Ollama interaction produce clear,
  actionable messages

## Standard Validation

Primary validation command:

```bash
pytest -q
```

Manual workflow validation with a real drive:

```bash
python archivist.py scan --drive /mnt/d/ --output reports/drive_d.csv
python archivist.py triage --csv reports/drive_d.csv
python archivist.py summarize --csv reports/drive_d.csv
python archivist.py organize --csv reports/drive_d.csv
python archivist.py copy --csv reports/drive_d.csv --dest /path/to/recovered
python archivist.py manifest --csv reports/drive_d.csv
```

Delete should only be exercised after manifest review:

```bash
python archivist.py delete --csv reports/drive_d.csv --manifest reports/delete_manifest.csv --confirm
```
